# 简单设备管理与资产分析边界 2026-07-10

## 状态

已接受。本文汇总近期设备部门、分析工作区、采集新鲜度和趋势展示的决策，并为后续开发设定产品边界。它补充而不改写历史记录；对于功能取舍，以本文的“停止扩张”原则为准。

## 产品定位

本插件是一个简单的设备管理和资产分析工具。它帮助管理员登记设备、接收客户端采集、查看资产状态和基础风险，并导出需要人工跟进的设备范围。

它不是 ITSM、工单、责任流、通知中心或远程运维平台。分析页的目标是让数据更直观、让设备资料更完整，而不是把管理员带入新的流程系统。

## 已完成的关键决策

### `未分配` 是不可删除的兜底部门

新建、导入、恢复或更新时没有有效部门的设备统一归入 `未分配`。该部门始终保留，不能从设置中删除，避免出现没有归属或失效部门的设备记录。

`未分配` 只保证数据可保存，不表示设备已经完成归属。硬件分析继续将它识别为 `部门待分配`，管理员可在设备资料或分析整改入口中补充真实部门。

### 分析使用事实数据，不把浏览器状态当作真相

设备异常由当前资产和最新采集数据派生。对于需要人工核查、但暂不修改设备事实的数据，`issue_handled` 和 `issue_reopened` 事件在服务端保存处理状态；刷新或更换管理员后仍保持一致。

部门、使用人等事实字段被修改后，检测器重新计算问题。修正事实与人工确认是两类动作，不能混为同一个“已处理”状态。

### 采集质量按年龄分级

硬件分析和管理摘要将采集时间按以下口径显示：

| 分级 | 含义 |
| --- | --- |
| `7 天内` | 当前采集正常 |
| `8-30 天` | 需要观察 |
| `31-60 天` | 逾期采集 |
| `61-90 天` | 严重逾期 |
| `超过 90 天` | 长期未更新 |
| `未接入` | 没有可用采集记录 |

三个逾期分级和未接入设备可按部门导出 CSV。导出只提供范围和资料，不会修改设备或创建任务；CSV 单元格统一处理，避免可编辑资料被表格软件作为公式执行。

### 分析仪表盘只保留必要信息

分析工作区保留三个面向设备管理的视图：

- 管理摘要：资产、异常和下一步概览。
- 硬件健康：采集质量、硬件库存和异常明细。
- 资产价值：金额、折旧和分布。

管理摘要显示近 30 天的采集接收量和问题状态变化。采集量按客户端采集时间统计；问题状态净变化定义为“确认已核查 - 恢复待处理”。这不是历史待办总量快照，界面必须继续保留该说明，避免过度解读。

硬件页的“客户端接入”只跳转到已有的令牌和上传配置界面，帮助管理员完成客户端配置；它不引入远程控制、自动重采集或设备命令下发。

## 明确不做的内容

以下能力会把简单设备管理演变为流程平台，当前不纳入规划：

- 工单、待办状态机、整改审批、SLA 或升级规则。
- 部门负责人、客户端责任人和复杂的责任分派模型。
- 消息通知、日报周报、自动催办或告警中心。
- 远程控制设备、自动触发采集、命令编排或安装运维系统。
- 为了“闭环”新增多层工作流、看板或配置页面。

如果未来确实需要其中某一项，应先以独立产品或清晰的外部系统集成为前提重新评估，不能在本插件中渐进叠加。

## 后续开发准则

后续只接受两类改动：

1. **数据准确性**：修复导入、采集、部门归属、资产状态、估值和统计口径中的事实错误。
2. **界面易读性**：减少噪声、统一术语、优化筛选/导出和现有指标的展示，使管理员更快理解设备与资产数据。

提出新功能时，先回答：它是否直接改善设备资料或资产分析？是否可以在现有页面和数据模型中完成？若需要新角色、新状态机、新定时任务或新控制平面，默认不做。

## 当前实现与验证入口

- 采集分级逻辑：`vite-admin/src/utils/hardwareAudit.ts`。
- 分析趋势接口：`GET /npcink-device-inventory/v1/analysis/trends`，仅管理员可访问。
- 趋势聚合：观察记录和事件仓储的按日聚合方法；前端固定展示 30 天并补齐空日期。
- 回归检查：`npm run check:hardware-audit`、`npm run check:analysis-trends` 和 `npm run check:release`。

本地后台已验证桌面和移动视图、部门导出入口、客户端接入跳转以及趋势接口的真实响应。后续改动应继续通过发布门禁，并在涉及管理界面时完成一次实际后台冒烟。
