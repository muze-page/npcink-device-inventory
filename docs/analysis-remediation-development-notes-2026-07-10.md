# 分析整改闭环：历史总结与开发原则 2026-07-10

## 状态

已落实并随 `v2.8.0` 发布。本文是后续开发分析工作区、资产归属字段和发布流程时的决策上下文，不替代数据模型与版本验收记录。

## 这轮解决了什么

设备分析此前能够发现问题，但几个基础信息问题缺少稳定的整改闭环：

- 空部门没有统一落点，导入、恢复和新建设备可能留下不一致值。
- 问题“已处理”只存在浏览器本地状态，刷新或换管理员后会丢失。
- `部门待分配` 与 `责任人缺失` 能被发现，却不能从问题表直接、可审计地整改。
- 闲置资产没有使用人时被当作异常，混淆了“设备在用责任”和“库存归属”。

本轮把分析页定位为数据质量整改入口，而不是只读报表：发现问题、选择受影响资产、修改事实字段、记录事件、重新计算问题，形成一条完整链路。

## 已确定的业务语义

### `未分配` 是兜底部门，不是有效归属

`未分配` 必须始终存在且不能在设置中删除。它用于承接新建、导入、更新和备份恢复时没有具体部门的资产，避免出现无部门或失效部门值。

它同时必须被硬件分析识别为 `部门待分配`。换言之，兜底保证数据可存，分析提醒保证数据不会被误认为已完成归属。后续报表、筛选和整改入口都应继续把它视为数据质量问题，而不能当作一个普通业务部门。

### `ownerName` 当前表示在用设备的使用人 / 责任人

只有 `active`（在用）资产且责任人为空时，才产生 `责任人缺失`。`inactive`（闲置）资产可没有使用人，其归属仍由部门承担；不能为了消除告警随意填入人员、位置或部门名称。

如果制度要求闲置设备也必须有明确保管人，应新增独立的“保管人”字段和规则，不要复用 `ownerName`。这能避免“当前使用人”和“库存保管责任”在同一字段里互相覆盖。

### 问题处理状态与事实整改是两类动作

`issue_handled` / `issue_reopened` 事件用于记录无法立即修改事实字段、但已完成核查的分析问题。状态由服务端事件折叠得出，不能回退到浏览器本地存储作为真相来源。

部门或责任人批量分配属于事实整改：更新资产后，检测器应自动不再产生对应问题，同时保留 `bulk_updated` 审计事件。不要再额外把同一问题标记为已处理，否则会把“数据已修正”和“人工确认暂不修正”混为一谈。

### 采集提醒与重复排查以行动价值为准

采集新鲜度仍应完整展示，便于运营了解数据时效；但它不等于待办。`长期未采集` 只在达到 180 天后生成：180–269 天为中风险提醒，270 天及以上为高风险。少于 180 天（例如 82 天）只能出现在新鲜度统计或资产详情中，不能占用异常队列或“逾期清单”。`未接入采集` 仍是独立问题，不应被误作超期采集。

维护中是资产生命周期状态，不是异常；它只应在资产状态和维护筛选中展示，不进入待处理队列或要求“确认已核查”。只有未来定义了维护期限、预计恢复日期和逾期规则后，才能单独新增“维护超期”问题。

重复风险必须按可核查的重复组呈现。资产编号相同仍是高风险；主 MAC 地址必须同时命中，才可产生 `疑似重复设备`，并为组内每台资产生成各自可审计的问题键和稳定的重复组键。硬件/系统 UUID 相同但主 MAC 不同属于 `历史标识冲突`，应留在只读身份审计中，不能进入重复设备待办或自动合并。主板序列号相同是低可信度采集线索，不能单独生成重复设备待办；当它与主 MAC 同时命中时，只作为问题明细中的补充依据展示。确认核查仍按资产问题键记录，不能把一个成员的人工结论隐式扩展到整组。

### 电脑唯一性以主板指纹统一

`device_uuid_v1` 是电脑资产的唯一自动匹配标识：服务端从主板厂商、型号、序列号和硬件 UUID 重新计算，不信任客户端声称的 UUID。磁盘、内存、网卡等可替换部件不会改变它；更换主板即视为新电脑。历史 V1/V2/V3 标识仅作兼容回退或审计线索。

数据工具的“设备身份梳理”先预览所有最新采集快照，只把计算结果唯一且未被其他资产占用的 `device_uuid_v1` 写入身份表，并为每次写入保留 `identity_reconciled` 事件。冲突、资料不足和既有资产绝不自动合并、覆盖或删除；这不是又一套清理流程，而是一次可重复、可审计的统一升级。

## 实现模式

### 分析问题的来源

`detectHardwareIssues()` 根据当前资产事实派生问题。问题键由资产 UUID 和问题类型构成，因而可稳定关联处理事件；问题本身不在独立表中持久化。

持久化的仅是处理事件。`GET /npcink-device-inventory/v1/analysis/issue-states` 按问题键读取最新的 `issue_handled` 或 `issue_reopened` 事件，前端据此显示已处理状态。这让多管理员、刷新和重新登录都看到同一结论。

### 批量整改的最小安全单元

分析页的批量操作必须遵循以下顺序：

1. 只允许选择当前可见、未处理且类型匹配的问题。
2. 先展示受影响资产数量与编号预览，再要求明确目标值。
3. 对每台资产更新实际字段。
4. 为每台资产写入 `bulk_updated` 事件，载明来源、字段旧值/新值、问题键和本次选择数量。
5. 刷新资产、分析、事件和问题状态查询；由检测器决定问题是否消失。

现有来源标识为：

| 问题 | 实际字段 | 事件来源 |
| --- | --- | --- |
| `部门待分配` | `department` | `analysis_department_assignment` |
| `责任人缺失` | `ownerName` | `analysis_owner_assignment` |

部门目标值必须排除 `未分配`。责任人输入可以提示已有姓名，但仍允许输入新人员；服务端继续负责权限、清洗和保存。

### 估值不套用同值批量覆盖

估值基础缺失已提供逐台编辑入口。采购价和残值通常随资产而不同，直接增加“批量填同一金额”会带来较高误改风险。除非先定义可复核的统一规则与预览口径，否则应保持逐台补录队列，而不是为了界面一致性复制部门/责任人的批量模式。

## 验证与发布经验

这轮验证不只依赖构建成功：

- 硬件分析夹具覆盖在用空责任人、闲置空责任人和纯空白责任人。
- 问题状态夹具覆盖处理与恢复的事件折叠。
- 提交包门禁运行前端构建、Lint、PHPStan、PHPCS、备份恢复、公开查询、资产搜索和问题状态检查。
- 本地 WordPress 浏览器冒烟实际创建临时在用/闲置资产，验证规则、批量分配、事件审计和刷新后的状态；测试数据随后清理。
- `v2.8.0` 已推送 GitHub、打标签并发布；线上下载的插件包再次验证了插件版本、Stable tag 和包结构。

发布门禁已把 `check:issue-states` 纳入统一检查，后续修改问题状态逻辑不应绕过该脚本。

## 后续开发边界

- 不删除或重命名 `未分配` 兜底语义；若需要显示名称调整，先处理已有数据与分析规则兼容性。
- 不把分析处理状态重新放回 `localStorage`、内存状态或单管理员会话。
- 不为不同问题类型建立无约束的通用批量写入。每种整改动作都要明确可选范围、目标字段、预览、审计来源和回归验证。
- 不将闲置资产的保管责任塞回 `ownerName`；需要时以独立字段和独立问题类型建模。
- 插件功能变更优先走插件发布。桌面端没有行为、签名、更新契约或依赖变化时，不提升桌面版本；当前标签工作流仍会构建桌面制品，这是发布流程后续需要单独收敛的事项。
- 相关仓库发布使用 GitHub 和命令行 Git；不恢复 Gitee 发布或同步步骤。

## 建议的下一阶段

优先把这一整改模式用于“需要人为判断、但可以明确填写单一事实字段”的问题，并逐项落地、逐项验收。候选顺序如下：

1. 评估是否需要独立的闲置设备保管人模型；这是业务制度决策，不应直接由前端批量功能替代。
2. 将估值待补保持为逐台工作队列，先补齐编辑效率、资料来源和复核记录，再评估受规则约束的批量能力。
3. 为 CPU、内存、硬盘等硬件缺失问题设计“补充来源”与可信度规则，避免把人工录入和客户端采集结果当作同一可信等级。
4. 将 GitHub 标签发布流程拆分为插件优先与桌面发布两条路径，避免纯插件版本无谓重建和更新桌面制品。

每次扩展前都应先回答：它修改的是哪一个事实字段、为什么该字段可批量修改、如何回滚或审计、修正后检测器如何自动收敛。不能清楚回答时，应先补业务规则而不是新增按钮。
