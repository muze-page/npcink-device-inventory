# ADR-001：以主板指纹统一电脑身份，并将数据质量整改做成可审计闭环

## 状态

已被 ADR-003 取代，2026-07-15。本文仅保留历史决策背景。

## 背景与历史问题

本项目经历过多代采集身份规则：

- V1 使用 `MD5(硬件 UUID + 第一张网卡 MAC)` 推导 UUID。
- 早期 V3 会按单个硬件 UUID、系统 UUID 或序列号等信号匹配。
- 后续 V3 增加了硬件/系统信号与主 MAC 的组合标识 `stable_device_id_v3`，降低了共享 SMBIOS UUID 的误合并风险。

新旧采集记录混存后，仍出现三个相互关联的问题：

1. 固件批量值、克隆镜像或历史数据会使多个资产拥有相同硬件 UUID；只看 UUID 会把不同电脑误判为重复或同一资产。
2. 主板序列号相同也可能是采集缺失、厂商默认值或历史脏数据，不能单独证明两条资产应合并。
3. “采集超期”“维护中”“疑似重复”等提醒若没有明确业务规则，会把日常状态噪声放进待处理队列，削弱真正问题的行动价值。

系统目标是保持简单：一台电脑需要一个可以持续识别的业务身份；历史资料需要能逐步统一，但任何不确定性都不能被自动合并或覆盖。

## 决策

### 1. 电脑唯一性以 `device_uuid_v1` 为准

`device_uuid_v1` 是当前唯一的电脑自动匹配主键。它由下列规范化信号计算：

```text
SHA-256(
  baseboard.manufacturer |
  baseboard.product/model |
  baseboard.serial |
  hardware_uuid
)
```

生成条件：必须有有效主板序列号，且必须再具备有效硬件 UUID，或同时具备主板厂商与型号。默认值、全零值和常见伪值不参与计算。

- 硬盘、内存、网卡、显卡等可替换部件不参与计算。
- 更换主板即生成新的 `device_uuid_v1`，业务上视为新电脑。
- 客户端可携带计算结果用于观测；服务端必须从上传快照重新计算，不能信任客户端声称的值。
- Rust 采集端与 PHP 服务端维护相同的固定夹具，保证跨端哈希结果一致。

### 2. 旧身份保留兼容，但不再定义新电脑

`legacy_device_id_v1`、`stable_device_id_v2`、`stable_device_id_v3`、硬件 UUID、系统 UUID、MAC 和其他序列号继续保留在身份表中，用于旧客户端上传、审计和迁移对照。

当上传能计算出 `device_uuid_v1` 时，只按它匹配，不回落到旧单信号身份；这样主板更换不会被归入旧电脑。当无法计算主板指纹时，才使用历史 V3/V2/V1 兼容路径，并继续保留 UUID 与 MAC 不一致时不合并的护栏。

这不是删除历史，而是明确历史身份的角色：它们是兼容线索，不是新的业务真相。

### 3. 数据梳理只补写确定身份，不自动合并资产

管理员在“数据工具 → 设备身份梳理”中执行迁移：

1. 读取每台未删除资产的最新采集快照。
2. 用服务端算法重新计算候选 `device_uuid_v1`。
3. 显示 `ready`、`already`、`collision`、`insufficient` 四种结果。
4. 管理员在确认已备份后，显式确认写入。
5. 仅为 `ready` 的一对一结果写入身份表，并记录 `identity_reconciled` 事件。

`collision`、`insufficient` 和已有冲突永远不自动写入；不会改 `npcink_assets.uuid`，不会自动合并、覆盖或删除资产。该设计使迁移可重复执行、可审计、可暂停，而不是增加另一套不可控的“清库”系统。

相关接口：

- `GET /wp-json/npcink-device-inventory/v1/analysis/device-identity-reconciliation`
- `POST /wp-json/npcink-device-inventory/v1/analysis/device-identity-reconciliation`，请求体为 `{ "confirm": true }`

接口只对管理员开放。

### 4. 重复设备、历史冲突与设备状态分开处理

- 资产编号相同是高风险重复线索。
- “疑似重复设备”必须形成可核查的重复组，并同时标识当前资产和同组候选资产；不能只在一行中给出模糊文字。
- 主 MAC 同时命中时，才可进入重复设备待办。硬件/系统 UUID 相同而主 MAC 不同，归类为只读的“历史标识冲突”，不自动合并。
- 主板序列号单独相同只是一条低可信度线索，不单独生成重复设备待办。
- `maintenance` 是生命周期状态，不是异常；除非未来定义维护期限和逾期规则，否则不要求“确认已核查”。
- 采集时效完整展示但只在 180 天后进入异常：180–269 天为中风险，270 天及以上为高风险。少于 180 天的记录（例如 82 天）不应占用待办队列。

## 被排除的方案

### 仅用硬件 UUID 或主板序列号

这两种信号都存在批量默认值、固件复用和采集缺失的现实风险，会导致错误合并。它们继续保留为证据和审计维度，但不单独定义电脑。

### 继续用 MAC 作为电脑唯一性

网卡可能增加、更换或停用，MAC 更适合作为网络接口证据；把它放入电脑的唯一性定义，会把换网卡误判为换电脑。

### 一次性自动清理或合并旧资产

历史冲突包含真实重复、采集错误和设备更换等多种情形。自动合并会破坏资产编号、责任人、财务字段和审计历史，且难以可靠回滚。因此当前只补写确定身份，合并必须另行设计预览、确认、回滚和 `merged` 审计流程。

### 为每一种数据问题增加独立工具

系统追求简单。身份梳理复用既有身份表、最新采集快照、事件审计和数据工具入口；不新增第二套设备注册表、第二个资产主键或通用批量改写引擎。

## 结果与开发约束

- `npcink_assets.uuid` 仍是资产记录的公开标识；`device_uuid_v1` 是电脑采集归属的身份声明，二者不可混用。
- 身份表的 `(identity_type, identity_value)` 全局唯一约束是防止同一确定身份指向多条资产的最后保护线。
- 所有自动或管理员写入都必须通过事件留下来源、时间和结构化载荷。
- 管理端预览不得把“可写入”表述为“已经合并”；冲突要明确提示人工核查。
- 新身份算法如需演进，必须新增版本（例如 `device_uuid_v2`），不得重解释既有 `device_uuid_v1`。
- 未来若主板信息在某个平台持续缺失，应先改善采集质量或定义人工补录可信度，再放宽唯一性条件。

## 验证基线

每次修改身份或整改逻辑至少验证：

- PHP 身份兼容夹具：主板不变、可替换件变化不变；主板变化则 UUID 变化；PHP 结果与固定跨端样例一致。
- Rust 采集端单元测试：生成合同字段且与 PHP 固定样例一致。
- 前端硬件审计夹具：共享 UUID 但 MAC 不同时不进入重复待办。
- PHP 业务夹具、前端 lint/build 与 `git diff --check`。
