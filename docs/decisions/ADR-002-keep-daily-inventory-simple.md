# ADR-002：保持日常资产管理简单，将历史治理能力收敛为高级操作

## 状态

已接受，2026-07-10。

## 背景

ADR-001 已将电脑身份统一为主板指纹 `device_uuid_v1`，同时保留 V1/V2/V3 的兼容证据，并提供身份审计、身份梳理和备份恢复能力。这些能力解决了真实历史数据问题，但如果与日常资产录入、查询和表格导入并列展示，会让普通管理员误以为每项工作都需要理解 UUID、兼容冲突或恢复语义。

在本地开发站点的首次梳理中，170 台资产中有 94 台被安全补写统一身份；2 台保留为冲突，74 台因历史导入快照缺少有效主板序列号而未写入。该结果说明治理能力是必要的，但也说明它不是日常台账操作。

与此同时，前端的单一 `pages/index.tsx` 已超过 8000 行。继续把所有流程堆在同一工作区，会同时提高产品理解成本、修改风险和测试成本。

## 决策

### 1. 日常与高级治理分层

“数据工具”默认只展示两项日常动作：

- 资产表格导入。
- 资产表格导出。

下列能力统一放入默认折叠的“高级治理与恢复”：

- 身份兼容审计。
- 设备身份梳理。
- 补齐采集信息（跳转既有客户端接入页）。
- JSON 备份导出与导入。

这不是权限变化：上述操作仍遵循原有管理员权限、确认和审计规则；只是明确它们的使用频率和风险等级。普通管理员不需要为了查询或导出资产而看到迁移和恢复操作。

### 2. 高级流程仍可回到日常资产上下文

身份兼容审计中的资产链接必须继续可用。点击后，管理端切换至“电脑资产”并打开对应详情，避免用户在治理页面中失去资产编号、责任人和状态等日常事实上下文。

### 3. 不新增自动合并，也不新增第二套补采流程

冲突资产和采集信息不足资产不会自动合并、覆盖或删除。系统只提供到既有“客户端接入”的入口：管理员让设备使用当前采集器重新上传，再回到身份梳理复核。

这一原则优先保护资产编号、归属、财务字段和审计历史。只有人工合并持续成为被证实的高频工作，且能够提供预览、保留源记录、回滚及 `merged` 审计事件时，才可以另立 ADR 设计合并能力。

### 4. 前端按高级能力先拆分

将身份兼容审计和设备身份梳理弹窗移动到 `components/advanced-data-governance.tsx`。该组件只依赖既有 REST 服务、类型和 UI 库；主页面保留工作区编排和跨工作区导航。

本次拆分不改变 REST 接口、身份算法或资产写入语义。后续若继续拆分，应优先选择边界明确的工作区或详情组件，避免为了文件行数而制造共享状态和循环依赖。

## 被排除的方案

### 删除高级能力

历史身份冲突、备份恢复和数据梳理已经证明有真实需求。删除它们会把风险转移为手工数据库操作或不可审计的临时脚本。

### 将高级能力继续放在硬件分析首页

硬件分析的目标是识别采集健康和硬件问题；身份兼容属于历史数据治理。混放会让异常队列、生命周期状态和迁移工具互相干扰。

### 立即继续拆完全部前端页面

全面重构会扩大回归面，偏离当前“停止新增功能、进入运营验收”的阶段目标。此时先抽离可独立验证的高级弹窗，后续仅在新的实际需求出现时按边界继续拆分。

## 运营停止条件与下一阶段入口

当前功能开发在此阶段停止。接下来的工作是运营验收：

1. 让 74 台历史导入设备重新使用当前采集器上传。
2. 人工核查 2 台冲突资产；未确认前不合并。
3. 再次运行身份梳理，观察 `ready`、`collision`、`insufficient` 的变化。

只有出现下列任一事实，才重新开启功能开发：

- 有效重采后仍存在大量无法生成主板身份的设备，需要改善采集端字段覆盖。
- 人工合并成为稳定、高频且可定义回滚规则的操作。
- 日常用户仍频繁误入高级治理，说明分层或文案需要继续调整。

## 验证基线

- 高级治理区默认折叠，日常数据工具不展示高风险按钮。
- 兼容审计可从高级治理打开，并能跳转到电脑资产详情。
- 硬件审计夹具、前端 lint/build、PHP 兼容与业务夹具、Rust 采集端测试均通过。
- 不因本次界面收口新增任何资产自动合并或数据库写入路径。
