# ADR-003：在首次真实使用前收缩产品范围与身份契约

## 状态

已接受，2026-07-15。

本决策取代 ADR-001 中“继续保留多代身份兼容与治理工具”的决定，也取代 ADR-002 中“仅折叠而不删除高级治理能力”的决定。两份旧 ADR 继续保留，作为项目在形成真实用户反馈前的历史记录。

## 背景

项目尚在开发阶段，没有真实用户、生产数据或向后兼容责任。现有实现已经同时包含：公开查询、多代设备身份兼容、身份审计与梳理、五种顶层资产类型、可写的问题处置状态、趋势和价值分析。这些能力各自可以解释，但尚未被真实使用验证，组合后显著扩大了产品理解、测试和维护成本。

已经通过验证且值得保留的基础包括：资产、身份、观测、事件四表模型；事务化上传；服务端身份计算；客户端令牌与 HMAC；备份恢复；发布门禁。主要问题不是基础设施失效，而是未经验证的产品范围过宽。

## 决策

### 1. 删除公开查询

删除公开短代码、公开 REST 路由、访问码和公开查询页面设置。资产查询只存在于已授权的 WordPress 管理端。

在出现经过验证的外部查询需求以前，不提供匿名、访问码或独立前台查询入口。

### 2. 身份契约只保留两条路径

电脑上传只使用下列身份之一：

1. `device_uuid_v1`：优先使用，由服务端从主板信息重新计算。
2. `fallback_device_v1`：仅在无法生成 `device_uuid_v1` 时使用，由服务端从一个有效硬件/系统序列信号与一个有效物理 MAC 重新计算。

两者都不能生成时，上传返回 `422`。服务器不信任客户端声明的哈希值。每次上传只为资产写入本次选中的主身份；硬件 UUID、系统 UUID、序列号和 MAC 保留在观测快照中，不再作为全局身份声明。

删除 `stable_device_id_v2`、`stable_device_id_v3`、`legacy_device_id_v1` 的上传兼容路径，以及身份兼容审计、自动梳理和冲突处置界面。项目没有需要迁移的真实用户数据，因此不引入双写、兼容窗口或后台迁移工作流。

### 3. 顶层资产类型只有 `computer` 与 `custom`

采集器上传创建 `computer`。非电脑资产统一使用 `custom`；网络、办公、移动等语义放入 `category`，不再扩张 `asset_type` 枚举。

开发数据库中的旧值按一次性 schema migration 归一：`pc` 和 `computer` 变为 `computer`，其他旧值变为 `custom`。

### 4. 分析中心只读

分析中心仅展示日常核查所需的总览、筛选和问题列表。它不保存“已处理/重新打开”状态，不提供趋势、价值分析、批量整改或身份治理操作。

资产详情仍可保存采购、价值等资产事实；删除的是尚未验证的分析工作流，不是基础资产字段。

### 5. 保留已验证的数据边界

本次不重写四表数据模型、REST 框架、事务化写入、备份恢复、客户端运行时配置、HMAC 或发布测试。大文件和样板代码只有在继续妨碍真实需求时才拆分，不能仅以行数作为重构理由。

## 被排除的方案

### 继续隐藏而不删除高级能力

隐藏只能降低首次可见复杂度，不能减少接口、状态、测试和维护成本，也会让未来开发默认这些能力已经成为产品承诺。

### 保留旧身份以防未来需要

项目没有真实兼容对象。为假设中的未来客户端保留多代匹配，会增加误合并设备和契约分叉风险。需要新算法时应新增明确版本，而不是保留未使用的历史路径。

### 全面重写插件或数据库

现有核心边界已经有测试和发布门禁，全面重写会扩大风险，却不能验证产品价值。当前应删除范围，不应更换技术栈或重造可靠基础设施。

## 后果

- 管理员能更快理解产品：采集电脑、维护资产、查看只读问题。
- 前后端接口、状态和测试数量减少，后续需求更容易基于真实反馈增加。
- 开发数据库中的旧分类会被破坏性归一；这是有意行为，不提供回滚兼容层。
- 如果未来确认需要公开查询、问题处置工作流或高级身份治理，必须以真实使用场景、权限边界和验收指标重新立项，而不是恢复旧代码。

## 验证标准

- 代码库中不再注册公开查询短代码或公开 REST 路由。
- 上传身份只接受服务端计算的 `device_uuid_v1` 或 `fallback_device_v1`，无有效身份时返回 `422`。
- 顶层资产类型只产生 `computer` 或 `custom`。
- 分析中心没有写入问题状态、趋势或价值分析请求。
- PHP fixtures、前端 lint/build、Rust fmt/clippy/test、发布和 WordPress.org 提交门禁全部通过。

## 下一阶段入口

本轮完成后停止继续扩功能，先用真实环境完成一个小规模试用：安装插件、创建客户端令牌、上传若干台电脑、维护一条自定义资产并处理分析页发现的问题。只有试用暴露稳定、重复的问题，才开始下一轮功能或结构重构。
