## 旧数据整理与新结构导入

旧数据来源：

```text
/Users/muze/gitee/npcink-device-inventory/设备数据备份
```

处理目标不是恢复旧插件结构，而是把旧数据整理成当前 v3 资产台账可用的数据，再导入当前站点预览。

本轮确认的迁移原则：

- 旧数据只作为一次性整理来源，不继续在产品内保留复杂的旧数据映射工作流。
- 新结构以资产编号、姓名、部门、状态、硬件摘要、财务字段和采集快照为中心。
- 管理员确认当前站点预览正常后，再从当前站点导出数据，导入到新部署站点插件中。
- 需要重点对比重复、遗漏和丢失，而不是只看导入数量。

旧数据中的硬件字段和新上传快照并不完全等价，因此展示层需要能同时读取：

- 标准资产字段
- 手动整理字段
- 旧数据导入字段
- 最新上传软件采集字段

## 搜索与资产识别

用户对比旧搜索结果和新搜索结果时发现，搜索 `32` 时旧系统能直观看到编号为 `32` 的电脑，新系统一开始排序和命中体验不一致。

本轮明确的搜索范围：

- 编号
- 名称
- 使用人
- 部门
- IP 地址
- MAC 地址

搜索范围扩大到 IP 和 MAC 是合理的，因为资产排查经常从网络线索开始；实现上应保持后端查询字段清晰，不把搜索做成难维护的自由全文匹配。

资产身份识别原则：

- 资产台账以插件端记录为准。
- 上传软件负责采集和上传，不负责编辑资产主数据。
- 上传时如果插件端已存在相关设备，更新设备信息后，向上传软件返回插件端的姓名和编号。
- 上传软件弹窗只展示必要确认信息，避免让普通用户理解过多管理字段。

关于重复上传：

- 上传软件的备注只在插件端姓名字段为空时才可作为补充。
- 如果插件端已经有姓名信息，上传软件备注不能覆盖插件端姓名。
- 这比让上传软件每次覆盖姓名更稳妥，因为管理员维护的台账信息优先级更高。

## 上传软件交互边界

上传软件定位为采集、上传和查看数据，不承担资产编辑职责。

本轮确定的用户提示策略：

- 上传成功后，弹窗显示已识别到的资产姓名和编号。
- 如果插件端已存在该设备，提示设备信息已更新。
- 如果插件端已有姓名或编号，上传软件只展示，不主动修改。
- 普通用户不需要看到部门、状态、匹配规则、重复风险等管理信息。

这个边界能避免上传端和插件后台形成两个编辑入口，也能减少误覆盖。

## 电脑资产列表体验

电脑资产列表从早期偏大的卡片逐步调整为更适合密集浏览的工作界面。

已确定的卡片原则：

- 卡片最大尺寸控制在约 `200 x 250`。
- 卡片内容优先展示使用人/名称、主板简写、CPU 简写、显卡简写、内存/硬盘、编号、状态、部门。
- 详细硬件型号进入详情弹窗查看，列表只展示可扫描摘要。
- 主板型号可去掉厂商和营销前缀，例如 `TUF GAMING B760M-PLUS WIFI D4` 在卡片中展示为 `B760M-PLUS WIFI D4`。
- CPU 可从 `Core(TM) i5-14600K` 简化为 `i5-14600K`。
- 显卡可从 `NVIDIA GeForce RTX 5060` 简化为 `RTX 5060`。

视图模式调整：

- 原“舒展卡片”改为“舒展模式”。
- 舒展模式只控制数据展示宽度，不改变卡片语义。
- 紧凑模式下页面最大宽度可收敛到约 `1100px`，提高扫描稳定性。
- 舒展/紧凑模式写入浏览器本地，刷新后不丢失。

排序规则：

- 默认排序应按最新上传或最新采集靠前，而不是沿用旧导入顺序。
- 如果展示顺序看起来仍像旧顺序，应检查后端 `sortBy` 参数、REST 查询默认值、前端请求参数和分页状态是否一致。

## 详情信息层级

用户反馈详情弹窗内硬件信息“字都一样大、一样重”，导致分不清主次。

本轮调整方向：

- 字段标题应比字段值弱，避免每个格子都像标题。
- 关键值可以更醒目，但不应和标题同权重。
- 硬件详情应以“标题、摘要值、补充说明”的层级展示。
- 长硬件文本、UUID、显示器信息要换行并保持可读。

这个问题不只出现在详情弹窗，也要在分析页、盘点页、卡片列表中检查。凡是密集信息区域，都应避免标题和值同字号、同字重、同颜色。

## 分析页调整

分析页当前包含硬件盘点和资产价值两个主要方向。

本轮对图表使用的判断：

- 排障和排名适合条形图，因为要比较 Top 项和定位异常。
- 占比结构适合饼图或环形图，例如电脑设备和自定义设备价值占比。
- 状态、部门、分类等如果类别不多，可以用饼图看比例；如果类别很多，应优先用条形图。
- 型号分布、硬盘型号、CPU 型号这类 Top N 排名不适合饼图，条形图更利于排查。

资产价值页已明确需要区分：

- 电脑设备价值
- 自定义设备价值
- 两者合计
- 两者占比

资产价值展示应同时保留金额总览和占比图，避免只给总数看不出结构。

## 硬件盘点与资产盘点

用户希望增加类似独立“资产盘点”的界面，用于从硬件维度快速看全局情况。

本轮确定的盘点界面方向：

- 放在分析或硬件盘点的合适位置。
- 上方用少量汇总块展示 CPU、硬盘、内存、主板数量。
- 下方根据所选类别展示盘点明细表。
- 点击上方类别或下方卡片可切换盘点口径。

字段口径：

- CPU 和主板可以看品牌、型号或具体型号。
- 内存只需要看容量，不需要看具体颗粒型号。
- 硬盘优先看容量，并可进一步区分机械硬盘和固态硬盘。

布局问题：

- 汇总块和表格之间需要合理间距，不能贴得太近。
- 表格标题、行内容和数字列需要清晰对齐。
- 表格区域不应出现过高的空白或无意义滚动。

## 后续建议

下一阶段不要继续扩大功能范围，应优先做真实部署后的验证：

- 在新部署站点安装 `v2.7.3` 插件包。
- 从当前站点导出确认过的数据。
- 导入新部署站点。
- 用上传软件在几台典型电脑上重新上传，确认同一台电脑能识别回原资产。
- 检查搜索编号、IP、MAC、姓名、部门是否符合预期。
- 检查资产价值、硬件盘点、详情弹窗和卡片模式是否符合日常管理习惯。

如果后续出现问题，优先按真实业务影响处理：

- 身份匹配和重复上传优先级最高。
- 数据遗漏和导入导出一致性其次。
- UI 密度和图表形式继续根据实际使用反馈微调。
- 不建议在发布后立即大改数据模型或上传软件职责边界。
