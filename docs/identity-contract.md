# 设备身份契约

## 状态

已接受。适用于电脑采集上传和管理员手工修正；取代多代身份兼容契约。

## 唯一两种身份

- `device_uuid_v1`：首选身份。服务端从主板厂商、型号、序列号和硬件 UUID 重新计算；更换主板视为新电脑。
- `fallback_device_v1`：只在无法生成主板身份时使用。服务端按固定顺序选择第一个有效硬件 UUID、系统 UUID、系统序列号、主板序列号或 BIOS 序列号，再与第一个有效物理 MAC 组合计算。

两种值均使用规范化后的事实计算 SHA-256 前缀。客户端可以本地显示同一算法的结果，但服务端不读取客户端声明的身份哈希。

服务端与 Rust 客户端共用无效值 fixture，拒绝全零/全 `F` UUID、`N/A`、`invalid`、OEM 默认串和其他常见占位值。不能从保守过滤后的事实得到身份时，宁可返回 `422`，也不使用低质量固件值合并设备。

## 上传决策

1. 服务端能计算 `device_uuid_v1`，只按该值匹配或创建资产。
2. 否则服务端能计算 `fallback_device_v1`，只按该值匹配或创建资产。
3. 两者都不能计算时返回 `422 missing_identity`，不创建资产、观测或事件。
4. 每次成功上传只声明本次选中的一个主身份；硬件 UUID、系统 UUID、序列号和 MAC 仅存入观测事实。
5. 全局唯一键和事务化 claim 继续处理并发首传，冲突请求回到已拥有该身份的资产。

## 不再支持

`stable_device_id_v2`、`stable_device_id_v3`、`legacy_device_id_v1`、单独的 UUID/MAC/序列号身份声明、身份兼容审计和自动梳理接口均已删除。项目首次真实使用前不承担这些开发期契约的兼容责任。

管理员手工身份接口仍保留，但只接受 `device_uuid_v1` 或 `fallback_device_v1`。
